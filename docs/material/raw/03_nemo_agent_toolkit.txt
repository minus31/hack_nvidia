Step 1: config.yml 작성
yaml
Copy
# hello-agent/config.yml

# ── LLM 정의 ──────────────────────────────────────────────────
llms:
  nim_llm:
    _type: nim                              # NVIDIA NIM 공급자
    model_name: nvidia/nemotron-3-nano-30b-a3b  # 사용할 모델
    temperature: 0.0                        # 결정론적 응답 (낮을수록 일관성 높음)
    max_tokens: 512                         # 최대 응답 토큰 수
    chat_template_kwargs:
      enable_thinking: false                # 빠른 응답을 위해 thinking 모드 비활성화

# ── 도구(Function) 정의 ────────────────────────────────────────
functions:
  wikipedia_search:
    _type: wiki_search                      # 내장 Wikipedia 검색 도구
    max_results: 3                          # 반환할 최대 검색 결과 수

  current_datetime:
    _type: current_datetime                 # 현재 날짜/시간 반환 도구

  code_generator:
    _type: code_generation                  # 코드 생성 도구 (LangChain 기반)
    programming_language: "Python"          # 생성할 코드 언어
    description: "Python 코드를 생성합니다. 코드 관련 질문에는 반드시 이 도구만 사용하세요."
    llm_name: nim_llm                       # 코드 생성에 사용할 LLM
    verbose: true

# ── 워크플로우(Agent) 정의 ────────────────────────────────────
workflow:
  _type: react_agent                        # ReAct 패턴 에이전트
  tool_names:                               # 이 에이전트가 사용할 도구 목록
    - wikipedia_search
    - current_datetime
    - code_generator
  llm_name: nim_llm                         # 에이전트의 추론에 사용할 LLM
  verbose: true                             # 추론 과정 로깅 활성화
  parse_agent_response_max_retries: 2       # 응답 파싱 실패 시 재시도 횟수
config.yml 구조 해설
config.yml
├── llms          → 워크플로우에서 참조할 LLM 인스턴스들 (이름으로 참조)
├── functions     → 에이전트가 호출할 수 있는 도구들 (이름으로 참조)
└── workflow      → 에이전트 타입과 사용할 llm/tool 바인딩

_type 필드가 핵심입니다. 모든 컴포넌트는 _type으로 종류를 지정하고, 나머지 필드는 해당 타입의 설정값입니다. tool_names와 llm_name은 위에서 정의한 인스턴스 이름을 참조합니다.

Step 1: 패키지 설정 (pyproject.toml)

NAT의 플러그인 시스템은 Python 패키지의 entry_points를 통해 커스텀 함수를 자동 등록합니다.

toml
Copy
# research-assistant/pyproject.toml
[build-system]
requires = ["setuptools>=64"]
build-backend = "setuptools.build_meta"

[tool.setuptools.packages.find]
where = ["src"]

[project]
name = "nat-research-assistant"
version = "0.1.0"
requires-python = ">=3.11"

[project.entry-points.'nat.components']
# 'nat.components' entry point에 등록된 모듈은 NAT가 시작 시 자동으로 import함
nat_research_assistant = "nat_research_assistant.register"

entry_points의 역할: nat run을 실행할 때 NAT는 nat.components 그룹에 등록된 모든 모듈을 자동으로 로드합니다. 이를 통해 YAML 내의 _type: unit_converter가 우리가 만든 Python 클래스와 매핑됩니다.

⚠️ 주의: 문서에는 nat.plugins 그룹이 언급되어 있으나, 실제 NAT 런타임이 스캔하는 그룹은 nat.components입니다. 공식 예제(simple_calculator)도 nat.components를 사용합니다.

참고: simple_calculator 예제의 register.py가 이 패턴의 공식 레퍼런스입니다.

Step 2: 커스텀 함수 구현 (register.py)

단위 변환 도구를 Python으로 구현하고 NAT 플러그인 시스템에 등록합니다.

python
Copy
# src/nat_research_assistant/register.py

from collections.abc import AsyncGenerator

from pydantic import Field

from nat.builder.builder import Builder
from nat.builder.function import FunctionGroup
from nat.cli.register_workflow import register_function_group
from nat.data_models.function import FunctionGroupBaseConfig


# ① Config 클래스: YAML의 _type: unit_converter 와 매핑되는 설정 모델
#    name="unit_converter"가 YAML의 _type 값이 됨
class UnitConverterConfig(FunctionGroupBaseConfig, name="unit_converter"):
    include: list[str] = Field(
        default_factory=lambda: ["length", "weight", "temperature"],
        description="포함할 변환 카테고리 목록"
    )


# ② @register_function_group 데코레이터로 NAT에 이 함수 그룹을 등록
@register_function_group(config_type=UnitConverterConfig)
async def unit_converter(
    _config: UnitConverterConfig, _builder: Builder
) -> AsyncGenerator[FunctionGroup, None]:
    """단위 변환 도구 그룹을 생성하고 등록합니다."""

    group = FunctionGroup(config=_config)  # ③ FunctionGroup 생성

    # ④ 각 도구 함수 정의: async def + docstring이 LLM에게 도구 설명으로 전달됨
    async def _convert_length(value: float, from_unit: str, to_unit: str) -> str:
        """길이 단위를 변환합니다.
        지원 단위: meter(m), kilometer(km), mile(mi), foot(ft), inch(in), centimeter(cm)
        예시: value=1.0, from_unit="mile", to_unit="kilometer"
        """
        factors_to_meter = {
            "m": 1.0, "meter": 1.0,
            "km": 1000.0, "kilometer": 1000.0,
            "mi": 1609.344, "mile": 1609.344,
            "ft": 0.3048, "foot": 0.3048,
            "in": 0.0254, "inch": 0.0254,
            "cm": 0.01, "centimeter": 0.01,
        }
        from_key = from_unit.lower()
        to_key = to_unit.lower()
        if from_key not in factors_to_meter:
            raise ValueError(f"지원하지 않는 단위: {from_unit}")
        if to_key not in factors_to_meter:
            raise ValueError(f"지원하지 않는 단위: {to_unit}")

        result = value * factors_to_meter[from_key] / factors_to_meter[to_key]
        return f"{value} {from_unit} = {result:.4f} {to_unit}"

    async def _convert_weight(value: float, from_unit: str, to_unit: str) -> str:
        """무게 단위를 변환합니다.
        지원 단위: kg(kilogram), g(gram), lb(pound), oz(ounce), t(metric ton)
        예시: value=70.0, from_unit="kg", to_unit="lb"
        """
        factors_to_kg = {
            "kg": 1.0, "kilogram": 1.0,
            "g": 0.001, "gram": 0.001,
            "lb": 0.453592, "pound": 0.453592,
            "oz": 0.0283495, "ounce": 0.0283495,
            "t": 1000.0, "ton": 1000.0,
        }
        from_key = from_unit.lower()
        to_key = to_unit.lower()
        if from_key not in factors_to_kg:
            raise ValueError(f"지원하지 않는 단위: {from_unit}")
        if to_key not in factors_to_kg:
            raise ValueError(f"지원하지 않는 단위: {to_unit}")

        result = value * factors_to_kg[from_key] / factors_to_kg[to_key]
        return f"{value} {from_unit} = {result:.4f} {to_unit}"

    async def _convert_temperature(value: float, from_unit: str, to_unit: str) -> str:
        """온도 단위를 변환합니다.
        지원 단위: celsius(C), fahrenheit(F), kelvin(K)
        예시: value=100.0, from_unit="celsius", to_unit="fahrenheit"
        """
        from_unit = from_unit.lower().strip()
        to_unit = to_unit.lower().strip()

        # 먼저 섭씨로 변환
        if from_unit in ("c", "celsius"):
            celsius = value
        elif from_unit in ("f", "fahrenheit"):
            celsius = (value - 32) * 5 / 9
        elif from_unit in ("k", "kelvin"):
            celsius = value - 273.15
        else:
            raise ValueError(f"지원하지 않는 단위: {from_unit}")

        # 섭씨에서 목표 단위로 변환
        if to_unit in ("c", "celsius"):
            result = celsius
        elif to_unit in ("f", "fahrenheit"):
            result = celsius * 9 / 5 + 32
        elif to_unit in ("k", "kelvin"):
            result = celsius + 273.15
        else:
            raise ValueError(f"지원하지 않는 단위: {to_unit}")

        return f"{value} {from_unit} = {result:.2f} {to_unit}"

    # ⑤ 그룹에 함수 추가: name은 YAML tool_names에서 참조될 이름
    if "length" in _config.include:
        group.add_function(name="length", fn=_convert_length, description=_convert_length.__doc__)
    if "weight" in _config.include:
        group.add_function(name="weight", fn=_convert_weight, description=_convert_weight.__doc__)
    if "temperature" in _config.include:
        group.add_function(name="temperature", fn=_convert_temperature, description=_convert_temperature.__doc__)

    yield group  # ⑥ 반드시 yield로 그룹 반환

코드 핵심 포인트:

FunctionGroupBaseConfig를 상속하고 name="unit_converter"를 지정하면 YAML에서 _type: unit_converter로 참조 가능
각 도구 함수는 async def로 정의하며, docstring이 LLM에게 전달되는 도구 설명이 됩니다
group.add_function(name=..., fn=..., description=...)으로 함수 등록 — YAML에서는 unit_converter__length 형태로 참조됨

Step 4: 통합 config.yml

참고 예제: examples/agents/auto_memory_wrapper/configs/config_mem0.yml — auto_memory_agent 설정의 공식 레퍼런스

Memory component가 필요한 이유 — NAT에서 대화 컨텍스트를 유지하는 방법은 두 가지입니다.

방법 A: 클라이언트 측 히스토리 (memory component 불필요)

messages 배열에 이전 대화를 직접 포함해서 보내는 방식입니다. NAT 서버는 별도 상태를 저장하지 않고, 요청마다 주어진 메시지 전체를 LLM에 전달합니다.

bash
Copy
curl -X POST http://localhost:8000/generate \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [
      {"role": "user",      "content": "파이썬의 GIL이 뭐야?"},
      {"role": "assistant", "content": "GIL은 ..."},
      {"role": "user",      "content": "그럼 우회 방법은?"}
    ]
  }'

단순 챗봇처럼 한 세션 안에서만 맥락을 유지하면 되는 경우 이 방법으로 충분합니다.

방법 B: Memory component (auto_memory_agent)

방법 A만으로는 해결되지 않는 상황이 있습니다.

한계	설명
세션 간 기억 불가	서버 재시작 또는 새 세션이 시작되면 이전 대화가 사라짐
컨텍스트 윈도우 한계	대화가 길어지면 LLM이 처리할 수 있는 토큰 수를 초과
관련 정보만 선별 불가	전체 히스토리를 그대로 넣어 노이즈 증가, 비용 상승
사용자 분리 불가	멀티테넌트 환경에서 사용자별 기억을 격리하기 어려움

auto_memory_agent는 이를 다음과 같이 해결합니다.

기능	클라이언트 히스토리	auto_memory_agent
세션 간 기억 유지	✗	✓ Mem0/Zep에 영구 저장
컨텍스트 윈도우 무제한	✗	✓ 관련 기억만 압축·검색
관련 정보만 주입	✗	✓ 시맨틱 검색으로 선별
사용자별 메모리 분리	직접 구현	✓ user_id 기반 자동 격리

메모리, 커스텀 함수, MCP 클라이언트를 모두 통합한 워크플로우 설정입니다.

yaml
Copy
# research-assistant/src/nat_research_assistant/configs/config.yml

# ── 전역 설정 ─────────────────────────────────────────────────
general:
  telemetry:
    logging:
      console:
        _type: console
        level: INFO    # DEBUG로 변경하면 더 상세한 로그 출력

# ── LLM 정의 ─────────────────────────────────────────────────
llms:
  nim_llm:
    _type: nim
    model_name: nvidia/nemotron-3-super-120b-a12b  # 메모리 처리에 충분한 크기의 Nemotron Super 모델
    temperature: 0.0
    max_tokens: 1024

# ── 메모리 백엔드 정의 ────────────────────────────────────────
# auto_memory_agent가 이 메모리를 사용해 대화를 자동으로 저장/검색함
memory:
  conversation_memory:
    _type: mem0_memory
    # MEM0_API_KEY 환경 변수에서 자동으로 인증 정보 로드

# ── 커스텀 함수 그룹 정의 ─────────────────────────────────────
# register.py에서 name="unit_converter"로 등록한 FunctionGroup과 매핑됨
function_groups:
  converter:
    _type: unit_converter
    include: [length, weight, temperature]  # 사용할 변환 카테고리

  # MCP 클라이언트: mcp-server-time 패키지를 stdio 방식으로 직접 실행
  # 별도 MCP 서버를 띄울 필요 없이 subprocess로 자동 관리됨
  mcp_time:
    _type: mcp_client
    server:
      transport: stdio                              # 로컬 프로세스로 실행
      command: "python"
      args: ["-m", "mcp_server_time", "--local-timezone=Asia/Seoul"]
    tool_overrides:
      get_current_time:
        # 도구 이름을 alias로 덮어쓸 수 있음 (충돌 방지)
        alias: get_current_time_kst
        description: "현재 한국 시간(KST)을 반환합니다. 현재 시각이 필요할 때 사용하세요."
    include:
      - get_current_time_kst  # alias를 include에 지정

# ── 개별 도구 정의 ────────────────────────────────────────────
functions:
  wikipedia_search:
    _type: wiki_search
    max_results: 3

  # inner_agent를 functions 섹션에 정의
  # auto_memory_agent가 이 에이전트를 inner_agent_name으로 참조함
  research_react_agent:
    _type: react_agent
    llm_name: nim_llm
    tool_names:
      - converter            # 커스텀 단위 변환 도구 그룹
      - mcp_time             # MCP에서 가져온 시간 도구 그룹
      - wikipedia_search     # 내장 위키피디아 검색 도구
    verbose: true
    parse_agent_response_max_retries: 3

# ── 최상위 워크플로우: 메모리 자동 래퍼 ─────────────────────
# auto_memory_agent는 inner_agent를 감싸서 메모리를 투명하게 처리함
# 사용자가 도구를 따로 호출하거나 프롬프트를 수정할 필요 없음
workflow:
  _type: auto_memory_agent
  inner_agent_name: research_react_agent  # 위에서 functions에 정의한 에이전트
  memory_name: conversation_memory        # 위에서 memory에 정의한 메모리 백엔드
  llm_name: nim_llm
  verbose: true
  description: "단위 변환, 위키피디아 검색, 시간 조회가 가능하며 대화를 기억하는 리서치 어시스턴트"

  # 메모리 동작 세부 제어 (모두 기본값 true)
  save_user_messages_to_memory: true      # 사용자 메시지를 메모리에 저장
  retrieve_memory_for_every_response: true # 매 응답마다 관련 기억을 검색
  save_ai_messages_to_memory: true        # AI 응답도 메모리에 저장

최종 아키텍처

LangGraph StateGraph로 구성한 멀티에이전트 워크플로우를 NAT가 감싸서 HTTP API / MCP / A2A 엔드포인트로 노출합니다. 코드 변경 없이 YAML만으로 올리는 방식(langgraph_wrapper)과, NAT 관리 LLM·옵저버빌리티를 완전히 활용하는 커스텀 등록 방식(@register_function) 두 가지를 지원합니다.

NAT × LangGraph 통합 방식
방식	언제 사용	코드 변경
langgraph_wrapper (내장 타입)	기존 LangGraph 에이전트를 그대로 NAT에 올릴 때	없음 (YAML만)
@register_function (커스텀 등록)	NAT 관리 LLM·도구·옵저버빌리티를 함께 사용할 때	Python 등록 코드 필요

NAT에서 도구·에이전트·워크플로우는 모두 "함수"로 관리됩니다. 커스텀 등록 시 FunctionBaseConfig + @register_function을 사용하며, name= 값이 YAML의 _type이 됩니다.

방식 A: langgraph_wrapper — 코드 변경 없이 기존 에이전트 올리기

기존에 동작하는 LangGraph 에이전트가 있다면 YAML 몇 줄로 NAT에 올릴 수 있습니다.

bash
Copy
uv pip install "nvidia-nat[langchain]" langgraph
yaml
Copy
# config.yml
llms:
  nim_llm:
    _type: nim
    model_name: nvidia/nemotron-3-nano-30b-a3b
    temperature: 0.0

workflow:
  _type: langgraph_wrapper
  dependencies:
    - ./my_langgraph_agent/   # sys.path에 추가할 패키지 경로
  graph: my_langgraph_agent.graph:compiled_graph  # module:변수명
  env: .env
bash
Copy
nat run --config_file config.yml --input "질문"
nat serve --config_file config.yml  # HTTP API 서버로 바로 배포

graph 필드는 LangGraph CLI의 --graph 인수와 동일한 형식입니다. compiled_graph는 graph.compile()로 만들어진 CompiledGraph 객체를 가리키는 변수명입니다.

방식 B: @register_function — NAT 통합 멀티에이전트 LangGraph

NAT가 LLM을 주입하고 옵저버빌리티·평가까지 연결되도록 LangGraph를 완전히 NAT에 통합합니다.

Step 2: register.py — LangGraph StateGraph + NAT 등록
python
Copy
# src/nat_lg_supervisor/register.py

from __future__ import annotations
from typing import Annotated, Literal
from collections.abc import AsyncGenerator

from pydantic import Field
from langchain_core.messages import HumanMessage, SystemMessage
from langgraph.graph import StateGraph, MessagesState, START, END

from nat.cli.register_workflow import register_function
from nat.data_models.function import FunctionBaseConfig
from nat.builder.function_info import FunctionInfo
from nat.builder.builder import Builder
from nat.builder.framework_enum import LLMFrameworkEnum
from nat.data_models.component_ref import LLMRef


# ① Config 클래스: name="lg_supervisor" → YAML의 _type 값
class LGSupervisorConfig(FunctionBaseConfig, name="lg_supervisor"):
    llm_name: LLMRef


# ② @register_function 데코레이터로 NAT에 등록
#    framework_wrappers=[LLMFrameworkEnum.LANGCHAIN] → NAT가 LangChain 호환 LLM 객체를 주입
@register_function(
    config_type=LGSupervisorConfig,
    framework_wrappers=[LLMFrameworkEnum.LANGCHAIN],
)
async def lg_supervisor(
    config: LGSupervisorConfig, builder: Builder
) -> AsyncGenerator[FunctionInfo, None]:
    """LangGraph 멀티에이전트 슈퍼바이저 워크플로우."""

    # ③ NAT Builder를 통해 LangChain 호환 LLM 획득
    llm = await builder.get_llm(
        config.llm_name, wrapper_type=LLMFrameworkEnum.LANGCHAIN
    )

    # ─ 노드 함수 정의 ──────────────────────────────────────
    async def supervisor_node(state: MessagesState) -> dict:
        """사용자 질문을 분석해 researcher 또는 coder로 라우팅."""
        sys_msg = SystemMessage(content=(
            "당신은 멀티에이전트 슈퍼바이저입니다.\n"
            "질문이 코드 작성에 관한 것이면 정확히 'ROUTE:coder'라고만 답하세요.\n"
            "정보 검색에 관한 것이면 정확히 'ROUTE:researcher'라고만 답하세요.\n"
            "다른 말은 절대 붙이지 마세요."
        ))
        response = await llm.ainvoke([sys_msg] + state["messages"])
        return {"messages": [response]}

    async def researcher_node(state: MessagesState) -> dict:
        """Wikipedia 등 외부 지식을 검색해 답변."""
        sys_msg = SystemMessage(content="당신은 정보 검색 전문가입니다. 질문에 대해 상세히 조사해 최종 답변을 작성하세요.")
        # 마지막 사용자 메시지만 전달 (supervisor 라우팅 메시지 제외)
        user_msgs = [m for m in state["messages"] if m.type == "human"]
        response = await llm.ainvoke([sys_msg] + user_msgs)
        return {"messages": [response]}

    async def coder_node(state: MessagesState) -> dict:
        """Python 코드를 생성해 답변."""
        sys_msg = SystemMessage(content="당신은 Python 전문 개발자입니다. 깔끔하고 잘 주석된 코드로 최종 답변을 작성하세요.")
        user_msgs = [m for m in state["messages"] if m.type == "human"]
        response = await llm.ainvoke([sys_msg] + user_msgs)
        return {"messages": [response]}

    def route_supervisor(state: MessagesState) -> Literal["researcher", "coder", "__end__"]:
        """'ROUTE:xxx' 패턴으로 다음 노드를 결정."""
        last = state["messages"][-1].content.strip()
        if "ROUTE:coder" in last:
            return "coder"
        elif "ROUTE:researcher" in last:
            return "researcher"
        return "__end__"

    # ─ StateGraph 구성: supervisor가 한 번 라우팅 → 워커 실행 → END ─
    #
    #   START → supervisor ──(ROUTE:researcher)──→ researcher → END
    #                      ──(ROUTE:coder)──────→ coder      → END
    #                      ──(그 외)────────────→ END
    #
    # 워커 노드가 supervisor로 돌아오지 않으므로 recursion_limit 오류 없음
    graph = StateGraph(MessagesState)
    graph.add_node("supervisor", supervisor_node)
    graph.add_node("researcher", researcher_node)
    graph.add_node("coder",      coder_node)

    graph.add_edge(START, "supervisor")
    graph.add_conditional_edges("supervisor", route_supervisor)
    graph.add_edge("researcher", END)  # 작업 완료 후 바로 종료
    graph.add_edge("coder",      END)  # 작업 완료 후 바로 종료

    agent = graph.compile()

    # ④ NAT가 호출할 최종 함수
    async def _run(input_message: str) -> str:
        result = await agent.ainvoke(
            {"messages": [HumanMessage(content=input_message)]}
        )
        return result["messages"][-1].content

    # ⑤ FunctionInfo.from_fn()으로 래핑해서 yield
    yield FunctionInfo.from_fn(_run, description=lg_supervisor.__doc__)

코드 핵심 포인트:

FunctionBaseConfig(name="lg_supervisor") → YAML _type: lg_supervisor에 매핑
builder.get_llm(..., wrapper_type=LLMFrameworkEnum.LANGCHAIN) → YAML llm_name이 가리키는 LLM을 LangChain 호환 객체로 주입
노드 함수들이 async def이므로 ainvoke로 비동기 실행
FunctionInfo.from_fn(_run) → NAT가 이 함수를 워크플로우로 인식

	langgraph_wrapper (A)	@register_function (B)
