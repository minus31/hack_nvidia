1. NemoClaw란?

NemoClaw는 OpenClaw AI 코딩 에이전트를 NVIDIA OpenShell 보안 런타임 위에서 실행하는 오픈소스 레퍼런스 스택입니다. AI 에이전트가 코드를 읽고 실행하는 과정에서 발생할 수 있는 보안 위협을 커널 수준에서 차단하면서도, 개발자 경험은 그대로 유지하는 것을 목표로 합니다.

구성 요소 개요
구성 요소	역할
OpenClaw	Anthropic Claude Code 기반의 AI 코딩 에이전트. 코드 읽기·쓰기·실행·git 조작·터미널 명령 등을 자율적으로 수행합니다. 오픈소스로 공개되어 있어 자체 호스팅 및 커스터마이징이 가능합니다.
OpenShell	NVIDIA가 개발한 AI 에이전트 전용 보안 런타임. Docker 컨테이너 위에 네트워크 정책 엔진, 자격증명 게이트웨이, 파일시스템 격리(Landlock LSM), 프로세스 제한(seccomp)을 레이어로 추가합니다. 에이전트가 허가되지 않은 외부 통신이나 파일 접근을 시도할 경우 커널 수준에서 즉시 차단합니다.
NemoClaw	OpenShell + OpenClaw를 단일 CLI로 통합한 레퍼런스 스택. 설치·프로비저닝·정책 설정을 자동화하며, 조직이 자체 보안 정책을 YAML로 선언해 적용할 수 있도록 합니다.
NemoClaw 스택 구조
NeMo Framework (모델 학습)
    └── Nemotron 모델 (오픈 웨이트 LLM)
            └── NIM (컨테이너 추론 서빙)
                    └── OpenShell (샌드박스 런타임 · 정책 엔진)  ← NemoClaw가 사용
                            └── NemoClaw (OpenShell + OpenClaw 통합 스택)
                                    └── OpenClaw (AI 코딩 에이전트)
OpenClaw 단독 사용 대비 NemoClaw의 이점

OpenClaw를 그대로 사용하면 에이전트가 호스트 머신에서 사실상 무제한의 권한으로 동작합니다. 코드 실행 중 임의 네트워크 요청을 보내거나, 프로젝트 범위 밖의 파일을 읽거나, API 키가 포함된 환경변수에 직접 접근할 수 있습니다. NemoClaw는 OpenShell을 통해 이러한 위험을 구조적으로 차단합니다.

항목	OpenClaw 단독	NemoClaw (OpenShell 포함)
네트워크 접근	호스트와 동일한 무제한 접근	deny-by-default + 허용 목록 정책, 핫 리로드 가능
API 키 관리	환경변수로 에이전트에 직접 노출	호스트 자격증명 저장소에 격리 → 에이전트 메모리에 전달되지 않음
파일시스템	사용자 홈 디렉토리 전체 접근 가능	Landlock LSM으로 허가된 경로만 접근
프로세스	권한 상승 및 임의 시스템 콜 가능	seccomp으로 허가된 시스템 콜만 허용
가시성	에이전트 동작 로그 없음	openshell term TUI로 실시간 활동 모니터링
Enterprise 환경에서의 강점

NemoClaw는 AI 코딩 에이전트를 사내 인프라에 안전하게 도입하려는 기업에 특히 적합합니다.

온프레미스 LLM 연동: OpenShell 게이트웨이가 추론 엔드포인트를 에이전트로부터 추상화하기 때문에, 사내 NIM 서버나 vLLM 클러스터를 외부 노출 없이 에이전트에 연결할 수 있습니다. API 키나 내부 주소가 에이전트 컨텍스트에 직접 나타나지 않아 내부 인프라 정보 유출 위험이 없습니다.
보안 정책 코드화: 허용할 네트워크 엔드포인트, 파일 경로, 외부 도구를 YAML로 선언하면 조직 전체에 동일한 정책을 일관되게 배포할 수 있습니다. 정책 변경은 재시작 없이 핫 리로드로 적용됩니다.
감사(Audit) 로그: 에이전트가 수행한 네트워크 요청, 파일 접근, 명령 실행 이력이 로깅되어 보안 감사·컴플라이언스 검토에 활용할 수 있습니다.
기존 인프라 통합: Docker 기반이므로 Kubernetes, CI/CD 파이프라인, 사내 컨테이너 레지스트리와 통합이 용이합니다. 개발자는 기존 OpenClaw 워크플로를 그대로 유지하면서 보안 레이어만 추가됩니다.
오픈소스·자체 호스팅: NemoClaw와 OpenClaw 모두 오픈소스로 공개되어 있어 벤더 종속 없이 사내 보안 요건에 맞게 커스터마이징할 수 있습니다.
NemoClaw가 해결하는 문제
문제	NemoClaw 해결 방식
에이전트의 무제한 네트워크 접근	기본 차단(deny-by-default) + 허용 목록 정책
API 키 유출	호스트에 격리 저장 → 에이전트 메모리에 전달 안 됨
파일시스템 탈출	Landlock LSM + seccomp 커널 수준 격리
운영 가시성 부재	openshell term TUI로 실시간 활동 모니터링
복잡한 배포 설정	단일 curl 명령으로 전체 스택 설치
2. 사전 요구사항
하드웨어

NemoClaw는 LLM 추론을 로컬에서 직접 수행하지 않고 NIM 엔드포인트(클라우드 또는 온프레미스)에 위임하므로, 클라이언트 머신에 GPU가 없어도 동작합니다. 아래는 에이전트 런타임 자체의 요구사항입니다.

	최소	권장	비고
CPU	4 vCPU	8+ vCPU	샌드박스 컨테이너 병렬 실행 시 여유 확보
RAM	8 GB	16 GB	8 GB 미만 시 샌드박스 프로비저닝 중 OOM 위험 — swap 설정 권장
디스크	20 GB	40 GB	샌드박스 이미지 ~2.4 GB, 프로젝트 워크스페이스 별도

온프레미스 LLM 추론을 함께 운영하는 경우 (NIM self-hosted), 추론 서버는 별도 GPU 노드에서 실행합니다. 참고 구성:

플랫폼	구성	용도
NVIDIA DGX Station / DGX H100	H100 80 GB × 4~8	팀 공유 추론 서버. NIM 컨테이너를 띄워 사내 NemoClaw 클라이언트들이 공동으로 사용하는 구성.
클라우드 GPU 인스턴스	AWS p4d/p5, GCP A3, Azure NDv5 등	온프레미스 GPU 없이 NIM을 클라우드에 self-hosted로 띄우거나, NVIDIA Build API(클라우드 NIM)를 직접 사용하는 구성.
NVIDIA Build API	GPU 불필요	로컬 GPU 없이 빠르게 시작하는 가장 간단한 구성. NVIDIA_API_KEY만 있으면 됨.
소프트웨어
항목	버전	비고
OS	Ubuntu 22.04 LTS+	macOS Apple Silicon, Windows WSL2도 지원 (Landlock LSM은 Linux 전용)
Node.js	22.16+	설치 스크립트가 자동 설치 시도
npm	10+	Node.js와 함께 설치됨
컨테이너 런타임	Docker Engine 24+	macOS: Colima 권장 / Windows: Podman (alias 불가, Docker 전용)
커널	Linux 5.13+	Landlock LSM 지원 필요 (Ubuntu 22.04 기본 충족)

4. Quick Start

참고 문서: docs.nvidia.com/nemoclaw/latest/get-started/quickstart.html

bash
Copy
# 1. 샌드박스에 연결
nemoclaw my-assistant connect

# 2. 터미널 UI로 채팅
# 최초 실행 시 디바이스 페어링 승인이 필요합니다:
#   openclaw devices list                    → 대기 중인 요청 ID 확인
#   openclaw devices approve <device-uuid>  → 해당 ID로 승인
openclaw tui

# 3. 단일 메시지 전송 (테스트)
openclaw agent --agent main --local \
  -m "안녕하세요, 현재 어떤 모델을 사용하고 있나요?" \
  --session-id test

# 4. 샌드박스 상태 확인
nemoclaw my-assistant status

전체 CLI 커맨드 목록은 7. CLI 레퍼런스를 참고하세요.

5. inference 프로파일 및 모델 변경

sandbox를 재시작하지 않고 런타임 중 모델을 변경할 수 있습니다. 공급자(provider) 자체를 바꾸려면 nemoclaw onboard를 재실행해야 합니다.

온보딩 시 입력한 API 키는 ~/.nemoclaw/credentials.json에 저장되며, 이후 추론 요청 시 OpenShell 게이트웨이가 자동으로 주입합니다.

현재 설정 확인
bash
Copy
openshell inference get
공급자별 모델 변경

참고 문서: docs.nvidia.com/nemoclaw/latest/inference/switch-inference-providers.html

공급자	provider ID	기본 모델
NVIDIA Endpoints	nvidia-prod	nvidia/nemotron-3-super-120b-a12b
OpenAI	openai-api	gpt-5.4
Anthropic	anthropic-prod	claude-sonnet-4-6
Google Gemini	gemini-api	gemini-2.5-flash
호환 엔드포인트	compatible-endpoint	직접 지정
bash
Copy
# NVIDIA Endpoints — 모델 변경
openshell inference set --provider nvidia-prod --model nvidia/nemotron-3-super-120b-a12b

# Anthropic
openshell inference set --provider anthropic-prod --model claude-sonnet-4-6

# OpenAI
openshell inference set --provider openai-api --model gpt-5.4

# 변경 후 확인
openshell inference get
로컬 모델 사용 (vLLM & Local NIM)

참고 문서: docs.nvidia.com/nemoclaw/latest/inference/use-local-inference.html

클라우드 API 대신 로컬에서 직접 모델을 서빙하면 데이터 프라이버시를 보장하고 네트워크 레이턴시를 최소화할 수 있습니다.

vLLM / nim-local 프로파일은 nemoclaw onboard 인터랙티브 메뉴에서만 선택 가능하며, 각각 아래 조건이 충족될 때만 메뉴에 표시됩니다.

vLLM: NEMOCLAW_EXPERIMENTAL=1 환경변수 + vLLM 서버가 localhost:8000에서 실행 중
nim-local: NEMOCLAW_EXPERIMENTAL=1 환경변수 + NIM-capable GPU 감지
vLLM

HuggingFace 모델을 OpenAI 호환 API로 서빙합니다. API 키 없이 완전 로컬 실행이 가능합니다.

vLLM 서버 시작
bash
Copy
pip install vllm

# 반드시 --host 0.0.0.0 으로 시작해야 컨테이너(sandbox)에서 접근 가능
vllm serve nvidia/Nemotron-3-Nano-30B-A3B --host 0.0.0.0 --port 8000
방화벽 설정 필요 (ufw 사용 환경): OpenShell sandbox는 Docker 네트워크를 통해 호스트의 vLLM에 접근합니다. ufw가 활성화된 경우 Docker 브리지 네트워크에서 포트 8000 접근을 허용해야 합니다.
bash
Copy
# OpenShell 클러스터 네트워크 허용
sudo ufw allow proto tcp from 172.18.0.0/16 to any port 8000 comment 'Allow OpenShell cluster to vLLM'
# Docker 기본 브리지 네트워크 허용
sudo ufw allow proto tcp from 172.17.0.0/16 to any port 8000 comment 'Allow Docker bridge to vLLM'

NEMOCLAW_EXPERIMENTAL=1 환경변수를 설정한 뒤 nemoclaw onboard를 실행하면 인터랙티브 메뉴에서 vLLM을 선택할 수 있습니다.

bash
Copy
NEMOCLAW_EXPERIMENTAL=1 nemoclaw onboard
# → 인터랙티브 메뉴에서 vLLM 선택
Local NIM

NVIDIA NIM 컨테이너를 로컬에서 실행하면 동일한 OpenAI 호환 API를 완전 로컬 환경에서 사용할 수 있습니다. 자세한 내용은 NIM 탭을 참조하세요.

NEMOCLAW_EXPERIMENTAL=1 환경변수를 설정한 뒤 nemoclaw onboard를 실행하면 NIM 이미지 pull 및 컨테이너 시작이 자동으로 처리됩니다.

bash
Copy
NEMOCLAW_EXPERIMENTAL=1 nemoclaw onboard
# → 인터랙티브 메뉴에서 "Local NVIDIA NIM [experimental]" 선택
# → NIM 이미지 자동 pull + 컨테이너 자동 시작
6. Telegram 연동

참고 문서: docs.nvidia.com/nemoclaw/latest/deployment/set-up-telegram-bridge.html

Telegram 브릿지를 활성화하면 Telegram 앱에서 직접 OpenClaw 에이전트와 대화할 수 있습니다. nemoclaw start는 브릿지와 함께 NemoClaw를 재기동하는 개념으로, 아래 절차대로 봇 토큰을 먼저 발급받아야 합니다.

Step 1 — Telegram 봇 생성

Telegram 앱에서 @BotFather 검색 후 대화 시작
/newbot 명령 입력
봇 표시 이름 입력 (예: My OpenClaw Assistant)
봇 사용자명 입력 — 반드시 bot으로 끝나야 함 (예: my_openclaw_bot)
BotFather가 봇 토큰을 발급 — 123456789:ABCDefgh... 형태로 표시됨

Step 2 — (선택) 내 Chat ID 확인

ALLOWED_CHAT_IDS를 설정하면 특정 계정만 봇에 접근할 수 있어 보안상 권장합니다.

Telegram에서 @userinfobot 또는 @getidsbot 검색
아무 메시지나 보내면 내 Chat ID(숫자)를 알려줌

Step 3 — cloudflared 설치 (Public URL 필요 시)

nemoclaw start 실행 시 "cloudflared not found — no public URL" 메시지가 나오면 Cloudflare Tunnel 클라이언트가 없는 것입니다. Telegram webhook은 HTTPS public URL이 필요하므로 아래와 같이 설치합니다.

bash
Copy
# Ubuntu (amd64)
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb -o cloudflared.deb
sudo dpkg -i cloudflared.deb

# Ubuntu (arm64)
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-arm64.deb -o cloudflared.deb
sudo dpkg -i cloudflared.deb

# 설치 확인
cloudflared --version

cloudflared는 Cloudflare 계정 없이도 임시 public URL(*.trycloudflare.com)을 생성할 수 있습니다. 계정 로그인 없이 바로 사용 가능합니다.

Step 4 — 토큰 등록 및 서비스 시작

bash
Copy
export TELEGRAM_BOT_TOKEN="123456789:ABCDefgh..."
export ALLOWED_CHAT_IDS="123456789"   # (권장) 허용 Chat ID — 복수 지정 시 쉼표로 구분
nemoclaw start

# 서비스 상태 확인
nemoclaw status

# 서비스 종료
nemoclaw stop

서비스 시작 후 Telegram에서 등록한 봇을 검색해 /start를 보내면 에이전트와 대화를 시작할 수 있습니다. ALLOWED_CHAT_IDS를 설정하지 않으면 봇 토큰을 아는 누구나 접근할 수 있으므로 주의하세요.

⚠️ nemoclaw start 전에 Telegram 네트워크 정책 프리셋이 적용돼 있어야 합니다. 미적용 시 브릿지가 api.telegram.org에 접근하지 못해 crash됩니다.

nemoclaw <name> policy-add telegram

⚠️ VPN 사용 중이면 api.telegram.org 접속이 차단될 수 있습니다. 브릿지가 시작되지 않을 경우 VPN을 끄거나 Telegram을 예외 처리한 뒤 재시도하세요.

명령어	설명

명령어	설명
openclaw tui	대화형 채팅 터미널 UI. 최초 실행 시 디바이스 페어링 필요
openclaw devices list	페어링 대기 중인 디바이스 목록 확인

openclaw agent --agent main --local -m "<메시지>" --session-id <id>	단일 에이전트 대화 턴 실행 (비대화형). 테스트·자동화에 적합
모델 및 추론 공급자 전환
명령어	설명

openshell inference update --model <model>	공급자 유지하고 모델만 변경
openshell provider list	등록된 공급자 목록 확인

공급자 추가·변경은 nemoclaw onboard를 통해 수행합니다. 공급자 등록은 온보딩 위저드가 자동으로 처리합니다.
